Imports vs declarations vs providers vs exports:-

-----

Imports: (to Import other modules)
Imports makes the exported declarations of other modules available in the current module. A template can use exported declarables from any imported module, including those from modules that are imported indirectly and re-exported.
For example, ModuleA imports ModuleB, and also exports it, which makes the declarables from ModuleB available wherever ModuleA is imported.
**Imports can only have NgModules.

declarations: (for components and Pipes)
Thare to make directives (including components and pipes) from the current module available to other directives in the current module. Selectors of directives, components or pipes are only matched against the HTML if they are declared or imported.

exports: ( allows components/directives/pipes in current modules to be imported using import from  ..)
exports makes the components, directives, and pipes available in modules that add this module to imports. exports can also be used to re-export modules such as CommonModule and FormsModule, which is often done in shared modules.


Providers: (for Services and Singleton instances)
They are to make services and values known to DI (dependency injection). They are added to the root scope and they are injected to other services or directives that have them as dependency.
When you provide the service at the root level, Angular creates a single, shared instance of service and injects into any class that asks for it. Other way to do this is by adding @Injectable decorator and specify providedIn property. Registering the provider in the @Injectable metadata also allows Angular to optimize an app by removing the service if it turns out not to be used after all.

	import { Injectable } from '@angular/core';
	@Injectable({   
	  providedIn: 'root', 
	}) 
	export class VoteService { }
